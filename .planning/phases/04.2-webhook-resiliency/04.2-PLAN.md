# Phase 4.2: Webhook resiliency - Plan

This plan hardens the Cashfree payment callback for high reliability, auditability, and transactional safety.

## Tasks

### 1. Database Schema Update [NEW]
- [ ] Create `03-webhook-logs.sql` to define the `webhook_logs` table.
- [ ] Columns: `id` (uuid), `cf_order_id` (text), `cf_payment_id` (text), `event_type` (text), `raw_payload` (jsonb), `status` (text), `created_at` (timestamp).

### 2. Idempotency Check [src/app/api/cashfree/webhook/route.ts]
- [ ] Check if the `cf_payment_id` has already been processed in the `orders` or `webhook_logs` table.
- [ ] Avoid duplicate fulfillment if the webhook is delivered multiple times.

### 3. Structured Logging [src/app/api/cashfree/webhook/route.ts]
- [ ] Log every incoming webhook request (after signature verification) to the `webhook_logs` table.
- [ ] Track whether the fulfillment was triggered successfully.

### 4. Error Handling & Atomicity
- [ ] Ensure DB updates use `select().single()` or `upsert` where appropriate.
- [ ] Return 200 OK to Cashfree even if fulfillment fails (to prevent infinite retries if the issue is on our end), but log the failure for manual recovery.

### 5. Verification
- [ ] Use `curl` to simulate a PAYMENT_SUCCESS_WEBHOOK.
- [ ] Verify logs in Supabase.
- [ ] Verify idempotency (second identical webhook should skip fulfillment).

## Verification Checklist

- [ ] Webhook log entry created for each request.
- [ ] Second delivery of the same payment ID does not trigger extra fulfillment.
- [ ] Full payload recorded in `webhook_logs`.
- [ ] High-density console logging for failures.
